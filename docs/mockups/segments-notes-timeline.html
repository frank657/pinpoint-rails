Showing segments & notes together

Segments are labelled time-ranges (chapters); notes are points or ranges that fall inside — or between — them. Three ways to make that relationship visible. Real data from Screen Recording 2026-06-04 (1:02): segment Test segment 0:36–0:44; notes at 0:00, 0:27–0:39, 0:36.

1 · Timeline track (under the player)

A shared time axis beneath the scrubber. Segments are bands; notes are pins (points) and bars (ranges) on a second lane. You instantly see which notes sit inside a segment — and which don't (the 0:00 intro note lives before any segment).

2 · Grouped outline (the side panel, reorganised)

Notes nest under the segment whose timeframe contains them; notes outside every segment collect in “Loose notes”. This is the clearest way to read “these notes belong to this chapter”.

Best for: reading & study. A note that straddles a segment edge (0:27–0:39 spills into the 0:36 segment) is shown by its start — or you can surface it in both with a faint “continues…” marker. “+ Add note” can live inside each segment header so new notes inherit the chapter.

3 · Combined: track + synced list recommended

The timeline track gives the spatial overview; the grouped list below it gives the detail. They’re synced — hover/seek in one highlights the other. This is the Snipd/Recall pattern adapted to your chapters.

Why this one: overview + detail without forcing a choice; the track answers “where” and the list answers “what”. Scales: when notes get dense the track clusters, the list stays readable. It also makes the segment↔note containment a real, shared model you can compute server-side (a note is “in” a segment when its start falls in the segment’s range).

Note on the data model: containment is derived, not stored — a note belongs to a segment when segment.start ≤ note.start < segment.end. Range notes that cross a boundary attach by their start (optionally shown as continuing). No schema change needed; it’s a grouping in the controller props.
